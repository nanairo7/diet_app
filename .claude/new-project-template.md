# 新プロジェクト向けテンプレート

diet_app の開発で確立したルール・コマンドを汎用化したものです。
新しいプロジェクトの `.claude/` フォルダにコピーして、
`【要変更】` の箇所を書き換えて使ってください。

---

## 1. CLAUDE.md テンプレート

プロジェクトルートの `CLAUDE.md` として配置します。

```markdown
# CLAUDE.md

## プロジェクト概要

【要変更】アプリの概要を1〜2文で書く。

## プラットフォーム・配信

【要変更】例:
- iOS / Android / Web など対象プラットフォーム
- 配信方法（TestFlight / Google Play / Web など）
- CI/CD ツール（Codemagic / GitHub Actions など）

## 技術スタック

【要変更】使用パッケージを記載する。

| カテゴリ | 技術 |
|---|---|
| フレームワーク | Flutter x.x / Dart x.x |
| 状態管理 | （例: Provider / Riverpod / Bloc） |
| 永続化 | （例: SharedPreferences / Hive / Isar） |
| テスト | flutter_test |

## ディレクトリ構成

【要変更】実際の構成に合わせて更新する。

\`\`\`
lib/
├── constants/   # 文字列定数・テーマ
├── models/      # データモデル
├── providers/   # 状態管理
├── screens/     # 画面
├── services/    # 外部連携・永続化
└── widgets/     # 再利用ウィジェット
test/            # テスト
\`\`\`

---

## ブランチ運用ルール

- バグ修正・機能追加は**必ず `main` の最新から新規ブランチを作成**すること
- 既存ブランチからの派生は禁止（差分混入を防ぐ）
- ブランチ命名規則: `feat/<機能名>` / `fix/<バグ名>`
- 作業完了後は必ず PR を作成すること（`main` への直接プッシュ禁止）
- PR は 1 機能 / 1 バグ修正につき 1 つ

## コミットメッセージ規約

| プレフィックス | 用途 |
|---|---|
| `feat:` | 新機能 |
| `fix:` | バグ修正 |
| `docs:` | ドキュメントのみの変更 |
| `refactor:` | 動作を変えないリファクタリング |
| `test:` | テストのみの変更 |
| `chore:` | ビルド・設定ファイルの変更 |

---

## 設計書の管理ルール

【要変更】設計書を管理する場合は記載する。不要なら削除。

- 新機能を実装したら `docs/design/` の該当ドキュメントも更新すること
- 画面の追加・変更時は画面フロー図を更新すること

---

## 開発ルール

- コミット前に必ず `flutter test` がパスすることを確認する
- 新しいウィジェット・画面を追加したら対応するテストも追加する
- 【要変更】テキスト文字列の管理方針（ハードコード禁止など）を記載する

---

## 既知の実装パターンと注意点

【要変更】開発中に発見したパターンや注意点を随時追記する。

### バグ調査・修正のフロー
- バグ修正を始める前に必ず原因調査を行い、報告すること
- 調査結果を確認してから修正に着手する（見当違いな修正を防ぐため）
```

## 2. カスタムコマンド テンプレート

`.claude/commands/` フォルダに配置します。

### `new-fix.md`

```markdown
以下の手順でバグ修正を行ってください。

1. `main` の最新から `fix/<バグ名>` ブランチを新規作成する
2. 関連ファイルを調査し、原因を特定して報告する（修正前に確認を取る）
3. 承認後、修正を実装する
4. 【要変更】静的解析・テストコマンドを実行する（例: `flutter analyze` → `flutter test`）
5. `fix:` プレフィックスでコミットする
6. PR を作成して URL を報告する

対象: $ARGUMENTS
```

### `new-feature.md`

```markdown
以下の手順で新機能を実装してください。

1. `main` の最新から `feat/<機能名>` ブランチを新規作成する
2. 実装方針・影響ファイルを整理して報告する（実装前に確認を取る）
3. 承認後、実装する
   - 【要変更】プロジェクト固有のコーディング規約を記載する
   - 新画面・ウィジェットには対応するテストを追加する
4. 【要変更】静的解析・テストコマンドを実行する（例: `flutter analyze` → `flutter test`）
5. `feat:` プレフィックスでコミットする
6. 影響する設計書があれば更新する
7. PR を作成して URL を報告する

機能概要: $ARGUMENTS
```

## 3. .gitignore への追記

`.gitignore` に以下を追加してください。

```
# Claude Code
.claude/worktrees/
.claude/plans/
```

## 4. 流用できるもの・できないものの整理

| 項目 | 流用可否 | 備考 |
|---|---|---|
| ブランチ運用ルール | ✅ そのまま使える | |
| コミットメッセージ規約 | ✅ そのまま使える | |
| バグ調査→修正フロー | ✅ そのまま使える | |
| `/new-fix` コマンド | ✅ ほぼそのまま | テストコマンドだけ書き換える |
| `/new-feature` コマンド | ✅ ほぼそのまま | コーディング規約部分を書き換える |
| `.gitignore` 追記 | ✅ そのまま使える | |
| 技術スタック | ❌ 書き直し | プロジェクトごとに異なる |
| ディレクトリ構成 | ❌ 書き直し | プロジェクトごとに異なる |
| 既知の実装パターン | ❌ 書き直し | 開発しながら随時追記する |
| プラットフォーム・配信情報 | ❌ 書き直し | プロジェクトごとに異なる |
